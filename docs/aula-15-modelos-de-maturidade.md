
# Aula 15 – Modelos de Maturidade

## Introdução

Este documento apresenta uma análise da maturidade do processo de desenvolvimento utilizado pela equipe no projeto LocalEats. A avaliação foi realizada com base nos conceitos de maturidade de processos estudados em aula, considerando aspectos relacionados à organização, padronização, gerenciamento e melhoria contínua das atividades de desenvolvimento.

O objetivo é identificar o nível atual de maturidade do processo da equipe, apontar lacunas existentes e propor melhorias que contribuam para a evolução da qualidade do processo e, consequentemente, do produto desenvolvido.

## 1. Diagnóstico de Maturidade

### Avaliação do Processo

| Critério                                                                             | Avaliação |
| ------------------------------------------------------------------------------------ | --------- |
| Os requisitos são documentados?                                                      | Sim       |
| Existe controle de mudanças?                                                         | Sim       |
| Há atividades de teste definidas?                                                    | Sim       |
| Os defeitos são registrados?                                                         | Parcial   |
| O processo de desenvolvimento é conhecido por toda a equipe?                         | Sim       |
| As tarefas são planejadas e acompanhadas regularmente?                               | Sim       |
| Existe padronização para implementação de funcionalidades?                           | Sim       |
| Os testes são executados antes da entrega das funcionalidades?                       | Sim       |
| Há revisão de código ou validação por outro integrante da equipe?                    | Parcial   |
| A equipe utiliza ferramentas para gerenciamento das atividades?                      | Sim       |
| Os artefatos do projeto (requisitos, testes e código) são organizados e versionados? | Sim       |
| Existe rastreabilidade entre requisitos e funcionalidades implementadas?             | Parcial   |
| A equipe realiza reuniões ou momentos de retrospectiva para identificar melhorias?   | Parcial   |
| Existem indicadores ou métricas para acompanhar a qualidade do projeto?              | Não       |

### Classificação do Processo

**Nível de Maturidade: Definido**

### Justificativa

O processo de desenvolvimento da equipe pode ser classificado como **Definido**, pois existe um fluxo de trabalho padronizado e conhecido por todos os integrantes. As atividades são organizadas utilizando o Trello, os requisitos são analisados antes do desenvolvimento, os protótipos são elaborados no Figma e o código é versionado por meio do Git. Além disso, há etapas de testes e correções antes da entrega das funcionalidades.

Embora ainda existam oportunidades de melhoria, como a formalização das revisões de código, a definição de métricas de qualidade e um controle mais estruturado dos defeitos, o processo já apresenta características de padronização, documentação e organização compatíveis com o nível Definido dos modelos de maturidade.

## 2. Identificação de Lacunas

| Lacuna                                                     | Impacto                                                                               |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| Ausência de métricas de qualidade                          | Dificulta acompanhar a evolução do projeto e medir a qualidade do processo.           |
| Revisões de código não formalizadas                        | Possibilita que erros passem despercebidos durante o desenvolvimento.                 |
| Registro de defeitos pouco estruturado                     | Dificulta a análise de falhas recorrentes e a priorização de correções.               |
| Rastreabilidade parcial entre requisitos e funcionalidades | Torna mais difícil verificar se todos os requisitos foram implementados corretamente. |

## 3. Propostas de Melhoria

| Melhoria                                                            | Benefício                                                                   |
| ------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Implementar revisões de código entre os integrantes da equipe       | Redução de erros e aumento da qualidade do código desenvolvido.             |
| Definir métricas de qualidade do projeto                            | Permite acompanhar a evolução do processo e identificar pontos de melhoria. |
| Criar um processo formal para registro e acompanhamento de defeitos | Facilita a gestão de falhas e reduz retrabalho.                             |
| Melhorar a rastreabilidade entre requisitos e funcionalidades       | Aumenta o controle sobre o desenvolvimento e a validação das entregas.      |

## Conclusão

A análise realizada demonstra que a equipe possui um processo de desenvolvimento organizado e padronizado, utilizando ferramentas adequadas para gerenciamento de tarefas, prototipação e controle de versão. Essas práticas contribuem para uma melhor comunicação entre os integrantes e para a qualidade das funcionalidades desenvolvidas.

Entretanto, ainda existem oportunidades de evolução relacionadas à medição da qualidade, à formalização das revisões de código e ao gerenciamento de defeitos. A adoção das melhorias propostas permitirá aumentar a maturidade do processo, tornando-o mais eficiente, previsível e alinhado às boas práticas de desenvolvimento de software.
